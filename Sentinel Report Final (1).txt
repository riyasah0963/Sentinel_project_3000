Abstract
This report documents the design, development, and evaluation of Sentinel, a lightweight file integrity monitoring (FIM) system built entirely in Python. The core objective of Sentinel is to detect unauthorised or accidental modifications to files within a specified directory by leveraging the SHA-256 cryptographic hashing algorithm. The system allows users to establish a baseline snapshot of a directory at a given point in time and subsequently compare the live state of that directory against the stored baseline to identify any files that have been added, removed, or modified.
The project was managed using the SPIRIT development methodology, encompassing Specification, Planning, Implementation, Review, Iteration, and Testing. Task management was facilitated through Trello, enabling clear tracking of milestones and deliverables. A command-line interface (CLI) was developed using Python's argparse library. All data is stored in JSON format to maximise portability and human-readability.
Testing was conducted across a variety of scenarios including file addition, deletion, modification, and edge cases. The system achieved 37 tests passing (1 skipped on Windows) with 83% code coverage. The project also discusses the legal, social, ethical, and professional (LSEP) considerations relevant to a file monitoring system.

1. Introduction
1.1 Background and Motivation
In the contemporary digital landscape, the integrity of files and data stored on computer systems is a matter of paramount importance. File integrity refers to the assurance that data has not been altered, corrupted, or tampered with in an unauthorised manner. Threats to file integrity arise from a wide variety of sources: malicious actors exploiting system vulnerabilities to inject malware or ransomware, insider threats where employees deliberately alter critical records or configuration files, and unintentional changes caused by system failures, software bugs, or simple human error. Regardless of the root cause, undetected file modifications can have serious consequences ranging from system instability and data corruption to regulatory non-compliance and security breaches.
File Integrity Monitoring (FIM) has long been recognised as a critical component of a robust cybersecurity posture. The principle behind FIM is conceptually straightforward: establish a known-good baseline of a file system at a trusted point in time, and subsequently verify that the current state of the file system matches that baseline. Any deviation from the baseline  -  whether an addition, deletion, or modification  -  is flagged for investigation. This approach provides an auditable trail and an early warning mechanism for a wide range of threats.
Despite the availability of enterprise-grade FIM solutions such as Tripwire Enterprise, OSSEC, and Qualys FIM, these tools are often prohibitively complex and expensive for small organisations, individual developers, or educational contexts. This project, Sentinel, was conceived to fill that gap, providing a simple yet functional integrity monitoring capability implemented in clear, well-documented Python code that is fully transparent in its operation and freely available as open-source software under the MIT Licence.
1.2 Aims and Objectives
The primary aim of this project is to design, implement, and evaluate a functional file integrity monitoring system that demonstrates core principles of cryptographic hashing, file system interaction, and software engineering best practices. The following specific objectives guide the development:
 To implement a baseline-generation module that recursively scans a target directory, computes SHA-256 hashes for each file, and stores the results in a structured JSON format.
 To implement a comparison module that reads the stored baseline, re-hashes the current state of the directory, and produces a detailed report of all additions, deletions, and modifications detected.
 To develop a user-friendly command-line interface (CLI) using Python's argparse library, providing intuitive commands for all core operations.
 To ensure robust error handling and input validation throughout the system, ensuring graceful behaviour under edge cases such as missing directories, unreadable files, and corrupt baseline data.
 To apply the SPIRIT methodology and agile project management principles throughout the development lifecycle.
 To conduct comprehensive testing achieving at least 80% code coverage across all modules.
 To evaluate the system critically against stated objectives and identify directions for future development.
1.3 Scope and Limitations
Sentinel is scoped as a single-user, single-machine tool operating via a command-line interface on systems running Python 3.10 or later. It is designed to monitor static directories on demand rather than providing continuous or real-time monitoring. The system has been developed and tested on Windows 11 (Python 3.14.2). The current scope does not include graphical user interface (GUI) development, network-based monitoring, or monitoring of file system metadata such as permissions and ownership. These are identified as future work in Section 11.
The deliberate constraint of using only Python's standard library  -  with no third-party runtime dependencies  -  was a key design decision that ensures Sentinel can be deployed on any system with a Python interpreter without package management complexities. This decision necessarily forecloses some implementation options (such as using the watchdog library for real-time monitoring) but significantly improves the accessibility and portability of the tool for its intended audience.

2. Literature Review and Background
2.1 File Integrity Monitoring: Foundations and History
The concept of file integrity monitoring has its roots in early Unix security practices. The seminal tool Tripwire, first developed by Gene Kim and Eugene Spafford at Purdue University in 1992, introduced the paradigm of cryptographic hash-based integrity verification to a mainstream audience (Kim and Spafford, 1994). Tripwire computed message digests of critical system files and stored them in a secured database; periodic checks would recompute the digests and compare them to the stored values, alerting administrators to any discrepancies. This foundational approach remains largely unchanged in modern FIM tools, demonstrating the enduring effectiveness of the underlying security principle.
The importance of FIM has been formalised in a number of industry standards and regulatory frameworks. The Payment Card Industry Data Security Standard (PCI DSS) mandates FIM for all systems that store, process, or transmit cardholder data (PCI Security Standards Council, 2022). Similarly, the NIST Cybersecurity Framework identifies integrity verification as a core component of the Detect function (NIST, 2018). The widespread regulatory adoption of FIM reflects both its effectiveness and its practicality as a lightweight, non-invasive approach to security monitoring that can be layered onto existing systems without major architectural changes.
Oriyano (2014) categorises FIM approaches along two axes: monitoring frequency (periodic versus real-time) and the storage location of the baseline (local versus remote). Periodic FIM, as implemented by Sentinel, scans at user-triggered or scheduled intervals and is well-suited to environments where the overhead of continuous monitoring is undesirable. Real-time FIM, exemplified by tools such as OSSEC, uses kernel-level event hooks  -  inotify on Linux, FSEvents on macOS, and ReadDirectoryChangesW on Windows  -  to receive immediate notifications of file system events. Each approach entails trade-offs that must be considered in the context of the specific deployment environment.
2.2 Cryptographic Hashing Algorithms
At the heart of any FIM system is a cryptographic hash function. A cryptographic hash function takes an arbitrary-length input and produces a fixed-length output with the following key properties: determinism (the same input always produces the same output); pre-image resistance (it is computationally infeasible to recover the input from the output); and collision resistance (it is infeasible to find two different inputs that produce the same output) (Schneier, 2015). The avalanche effect, whereby even a single-bit change to the input produces a completely different output, is what makes cryptographic hashes so effective for detecting file modifications: a file that has been modified even minimally will produce an entirely different digest.
The MD5 algorithm is now considered cryptographically broken due to collision vulnerabilities demonstrated by Wang and Yu (2005), who showed that two distinct files could be crafted to produce the same MD5 hash. SHA-1 was similarly deprecated following the SHAttered attack (Stevens et al., 2017), which produced the first practical full SHA-1 collision. Sentinel uses SHA-256, part of the SHA-2 family standardised by NIST (FIPS PUB 180-4, 2015). SHA-256 produces a 256-bit digest and has not been subjected to any successful collision or pre-image attacks as of 2025, making it the appropriate choice for contemporary integrity monitoring applications.
It is important to note that hash comparison alone cannot detect replay attacks, in which an attacker who has write access to both the monitored files and the baseline file restores the original file content after a period of malicious activity. More sophisticated integrity systems combine hashing with digital signatures, trusted timestamping, and immutable audit logs to guard against this class of attack. For the scope of Sentinel, SHA-256 hashing is considered sufficient for the educational and lightweight production use cases targeted by the tool.
2.3 Existing FIM Solutions
A survey of existing FIM tools reveals a spectrum from complex enterprise platforms to lightweight open-source utilities. Tripwire Enterprise remains an industry leader offering centralised management, role-based access control, and SIEM integration (Tripwire, 2023). OSSEC, an open-source HIDS, includes FIM capabilities alongside log analysis, rootkit detection, and active response (OSSEC Project, 2023). Wazuh provides a comprehensive security platform with a modern web-based dashboard and cloud integration. At the lighter end, tools such as aide (Advanced Intrusion Detection Environment) and samhain provide standalone FIM for Unix-like systems.
Sentinel belongs to the lightweight educational category, with its key differentiators being code transparency, portability across Windows, Linux, and macOS, and the absence of any runtime dependencies beyond a standard Python installation. The codebase is explicitly designed to be readable and well-documented so that the underlying security principles are immediately visible to students and practitioners seeking to understand how FIM systems work at an implementation level.
2.4 JSON as a Data Storage Format
The choice of JSON (JavaScript Object Notation) as the storage format warrants explicit justification. JSON is a lightweight, human-readable data interchange format formalised by Crockford (2006). Its hierarchical key-value structure maps naturally to a file system baseline, where each relative file path serves as a key and its SHA-256 hash and file size constitute the value. JSON files are portable across platforms, easily parsed by any modern programming language, version-controllable alongside source code, and human-inspectable without specialist tools, making it the optimal choice for Sentinel's requirements. Alternative formats such as XML are more verbose; binary formats such as SQLite would sacrifice human-readability; plain text formats would require a custom parser.

3. Legal, Social, Ethical and Professional Considerations
3.1 Legal Framework
The deployment of file integrity monitoring software raises important legal considerations. In the United Kingdom, the Computer Misuse Act 1990 prohibits unauthorised access to computer systems and the unauthorised modification of computer material. Sentinel is explicitly designed as a tool for monitoring files on systems that the user owns or has explicit authorisation to monitor. Deploying Sentinel on systems belonging to others without authorisation would constitute a criminal offence under Section 3 of the Act regardless of the monitoring intent.
The General Data Protection Regulation (GDPR), retained in UK law through the UK GDPR and the Data Protection Act 2018, imposes obligations on organisations that process personal data. If Sentinel is used to monitor directories containing personal data, the operator must ensure compliance with GDPR principles including lawfulness and transparency (Article 5(1)(a)) and data minimisation (Article 5(1)(c)). The baseline JSON files may themselves constitute personal data if they contain file paths that identify individuals, and must therefore be stored securely and subject to appropriate access controls and retention policies.
The MIT Licence under which Sentinel is released permits use, modification, and redistribution with minimal restrictions, while explicitly disclaiming liability. Export control regulations governing cryptographic software (EAR in the United States, UK Export Control Order 2008) do not impose additional obligations on Sentinel beyond those already applicable to Python itself, since SHA-256 is a widely published algorithm implemented in Python's standard hashlib library.
3.2 Ethical Considerations
The deployment of FIM in a workplace context raises concerns about employee surveillance if workers are not informed that their files and activities are being monitored. Transparency requires that all individuals whose files may be monitored are informed and have given appropriate consent, consistent with UK employment law requirements under the Regulation of Investigatory Powers Act 2000 (as amended). The dual-use nature of FIM tools deserves acknowledgement: while Sentinel's purpose is defensive integrity monitoring, the same capability could be misused for covert surveillance. This risk is addressed through documentation explicitly specifying intended use cases and prohibiting deployment without appropriate authorisation.
The project was conducted under the University of Plymouth's ethical research guidelines. No user data was collected or processed during development. All test data was synthetically generated. The use of Generative AI tools was limited to the permitted categories specified in the COMP3000 assignment brief (A2, A3, A5, A6, A7, A8, A9) and is transparently declared in Appendix A. AI-generated code suggestions were critically reviewed, understood, and in some cases modified before incorporation, consistent with the requirement that students be able to explain and justify all submitted solutions.
3.3 Professional Considerations
The development of Sentinel has been conducted in accordance with the British Computer Society (BCS) Code of Conduct (BCS, 2022), which requires members to act with integrity, to have due regard for the rights of others, and to uphold the standing of the profession. Specific relevant provisions include the requirement to develop and maintain professional knowledge and competence (addressed through the literature review), and the requirement to ensure that products are fit for purpose (addressed through comprehensive testing).
Version control using Git with a clean commit history, conventional commit messages, and a structured branching strategy demonstrates professional software development practice consistent with industry standards. The README.md follows the standard open-source project documentation structure, enabling Sentinel to function as a genuine open-source contribution rather than merely an academic submission. The code achieves PEP 8 compliance verified by flake8 and consistent formatting enforced by black.

4. Methodology  -  The SPIRIT Model
4.1 Overview of SPIRIT
The SPIRIT methodology is the structured software development framework adopted for this project. SPIRIT stands for Specification, Planning, Implementation, Review, Iteration, and Testing. It combines elements of traditional plan-driven development with the iterative and adaptive qualities of agile methodologies. This hybrid approach was judged appropriate for a project where the high-level requirements were relatively well understood at the outset but where iterative refinement based on testing feedback was expected throughout development.
Pure Scrum (Schwaber and Sutherland, 2020), while industry-standard, is designed for teams and its ceremony overhead is disproportionate for a solo project. The Waterfall model (Royce, 1970) was rejected because the problem domain required iterative discovery. SPIRIT provides a middle path: sufficient structure for academic documentation while preserving the flexibility to adapt scope and design as understanding deepens. Its six phases map naturally onto the COMP3000 assessment milestones while preserving an iterative and reflective character throughout.
4.2 Specification
The specification phase involved a thorough analysis of the problem domain and derivation of system requirements through literature review and analysis of existing FIM tools. The specification produced a set of user stories following the agile convention. Key stories included: `As a security-conscious developer, I want to create a hash baseline of my project directory so that I can detect any unauthorised changes to my codebase'; and `As a system administrator, I want to run periodic integrity checks so that I receive a clear report of any files that have changed since the last baseline was created'. These formed the initial product backlog prioritised using the MoSCoW framework.
4.3 Planning
The planning phase translated the product backlog into a concrete project schedule across three two-week sprints. Sprint 1 focused on baseline creation and JSON storage. Sprint 2 targeted comparison logic and CLI development. Sprint 3 addressed testing, documentation, and refinement. Trello was used as the primary task management tool with columns: Backlog, Doing, Done, User Story, and Tools and Technologies.

6. System Design and Architecture
6.1 Modular Architecture
Sentinel is designed according to a modular architecture separating concerns into five discrete, independently testable components: the Hashing Module (core/hasher.py), the Scanner Module (core/scanner.py), the Baseline Module (core/baseline.py), the Comparison/Checker Module (core/checker.py), and the CLI Entry Point (sentinel.py), supported by utility modules core/logger.py and core/utils.py. This separation follows the Single Responsibility Principle (SRP) from the SOLID design principles (Martin, 2003), ensuring each module has one clearly defined reason to change. The modular design significantly simplifies testing, as each module can be exercised in isolation using pytest's tmp_path fixture for file system isolation.
6.2 System Architecture Diagram
Figure 3 illustrates the high-level architecture and data flow of the Sentinel system, from user invocation through the CLI to the final comparison report output:
                                       
Figure 3: Sentinel system architecture showing data flow: User --> CLI --> File Scanner --> Hash Files --> Baseline (JSON) --> Compare --> Alert / No Changes Detected.
As shown in Figure 3, the user interacts exclusively with the CLI layer. The CLI invokes either the baseline creation workflow (left path: scan --> hash --> serialise to JSON) or the integrity check workflow (right path: load JSON --> scan --> hash --> compare --> report). The clean separation between these workflows ensures that the baseline creation and comparison operations are independently maintainable and testable.
6.3 Data Model
The baseline JSON file adheres to a well-defined schema. The top-level object contains: sentinel_version, schema_version, created_at (UTC ISO 8601 timestamp), base_path (absolute path of the monitored directory), total_files, skipped_files, elapsed_seconds, and a files object. Each entry in files maps a relative file path to an object containing the SHA-256 hash and the file size in bytes. Using relative paths ensures baselines remain valid even if the monitored directory is relocated, addressing a key portability requirement identified during the specification phase.

6.4 System Flow
In baseline creation mode, the CLI validates the target directory, invokes the Baseline Module, which uses os.walk() to traverse the directory tree recursively, passes each file path to the Hashing Module, and accumulates results in a dictionary keyed by relative path. The accumulated dictionary is wrapped in the metadata envelope and serialised to JSON with 2-space indentation for human readability. In comparison mode, the Comparison Module loads and validates the baseline JSON, traverses the current directory state, re-hashes all accessible files, and computes three categories of difference using set operations: deleted files (baseline keys minus current keys), added files (current keys minus baseline keys), and modified files (keys in the intersection whose hashes differ between baseline and current state).

7. Implementation
7.1 Development Environment
Development was conducted on a Windows 11 machine using Visual Studio Code as the primary IDE, with the Python, GitLens, and Pylance extensions providing IntelliSense, type checking, and version control integration. Python 3.11 was the primary development target, and the project was additionally verified to run correctly under Python 3.14.2 (the version present on the test machine at submission time). A dedicated virtual environment was created using python -m venv .venv to isolate project dependencies. The flake8 linter and black code formatter were configured as pre-commit hooks to enforce PEP 8 compliance and consistent formatting throughout the codebase.
7.2 The Hashing Module (core/hasher.py)
The hashing module implements the compute_sha256() function, which accepts a file path and returns the hexadecimal SHA-256 digest, or None if the file cannot be read due to permission restrictions or I/O errors. The function reads files in chunks of 65,536 bytes (64 KB), updating the hash object incrementally using the hashlib.sha256().update() interface. This chunked approach ensures arbitrarily large files can be hashed without exhausting system memory  -  only 64 KB is held in memory at any one time regardless of file size. Files are opened in binary mode (`rb') to ensure hash values are consistent across platforms regardless of line-ending conventions.
  def compute_sha256(file_path: str) -> Optional[str]:
      """Compute SHA-256 hash of a file using 64KB chunked reading."""
      try:
          h = hashlib.sha256()
          with open(file_path, "rb") as f:
              while chunk := f.read(65536):
                  h.update(chunk)
          return h.hexdigest()
      except (PermissionError, IOError) as e:
          logger.warning("Cannot hash %s: %s", file_path, e)
          return None

Exception handling catches specific PermissionError and IOError exceptions  -  a deliberate improvement over a bare except clause, which was identified as a code quality issue during Sprint 2 review. Catching specific exceptions prevents masking unexpected errors while still handling the expected case of encountering an inaccessible file during a directory scan.
7.3 Baseline Creation Module (core/baseline.py)
The baseline module implements the create_baseline() function, which accepts a directory path and output file path. It uses os.walk() to traverse the directory tree recursively, storing each file's hash keyed by its path relative to the monitored root. The use of relative paths rather than absolute paths is a critical design decision: it ensures that baselines remain valid if the monitored directory is moved, and prevents the baseline from containing machine-specific path prefixes that would break portability. Upon completion, the results are wrapped in the metadata envelope and serialised to JSON.

7.4 Comparison Module (core/checker.py)
The comparison module implements the check_integrity() function. It first loads and validates the stored JSON baseline file, verifying that the expected schema keys are present and that the base_path field is accessible. It then performs a fresh recursive scan of the current directory state using the same os.walk()-based traversal as the baseline creation module, computing SHA-256 hashes for all accessible files. Set operations then compute the three categories of change in O(n) time: deleted files are computed as the set difference of baseline keys minus current keys; added files as current keys minus baseline keys; and modified files as the intersection of both key sets where the stored hash differs from the freshly computed hash.
The results are displayed with ANSI colour coding: green for added files (indicating new files that were not present at baseline time), red for deleted files (files present at baseline that have since been removed), and amber for modified files (files present in both states but whose content has changed). For modified files, both the original baseline SHA-256 hash and the current SHA-256 hash are displayed side by side, providing an auditable record useful for forensic investigation. ANSI colour output is automatically disabled when stdout is not connected to a terminal (TTY), ensuring clean output when the tool is used in automated scripts or CI/CD pipelines.
7.5 Command-Line Interface (sentinel.py)
The CLI uses Python's argparse library, providing two main operational modes: --init (create a new baseline) and --check (perform an integrity check against an existing baseline), with optional flags --verbose (list all files processed during the scan), --report (save a structured JSON report to a specified path), and --version (display the tool version). The system exits with Unix-standard exit codes: 0 on clean completion with no changes detected, 1 on completion with changes detected or on execution errors, enabling transparent integration with shell scripts and CI/CD pipeline conditional logic. The ASCII art Sentinel banner displayed on startup confirms the tool version and institutional context for the COMP3000 submission.
7.6 Error Handling and Code Quality
The system handles all common error conditions with descriptive messages: the target directory does not exist; the baseline file is missing or its path is incorrect; the baseline file contains malformed JSON; individual files cannot be read due to permission restrictions (scan continues, file is skipped with a warning); and the output directory is not writable. All functions include Google-style docstrings documenting parameters, return values, and exceptions. The codebase achieves PEP 8 compliance verified by flake8 and consistent formatting enforced by black. Inline comments explain non-obvious logic, and module-level docstrings document the design decisions motivating each component's implementation approach.

11. Future Work
11.1 Real-Time Monitoring
The most significant functional gap is the absence of real-time monitoring. Real-time capability would require integration with OS file system event APIs: inotify on Linux, FSEvents on macOS, and ReadDirectoryChangesW on Windows. Python's watchdog library provides a cross-platform abstraction over these APIs and would be the natural implementation choice for a future real-time mode, enabling Sentinel to detect changes immediately rather than only during scheduled or user-triggered scans. The modular architecture of the current implementation provides a strong foundation: a new realtime subcommand would reuse the existing hasher and baseline modules as building blocks.
11.2 Graphical User Interface
A GUI would significantly improve accessibility for users unfamiliar with command-line tools. A desktop GUI built using Python's tkinter standard library would require no additional dependencies and is consistent with the current zero-dependency philosophy. Alternatively, a web-based interface using Flask or FastAPI would offer a more modern user experience, facilitate remote monitoring of multiple directories, and enable integration with notification services such as email or Slack webhooks for automated alerting when changes are detected.
11.3 Metadata and Permission Monitoring
Extending Sentinel to monitor file metadata alongside content hashes would increase its security value significantly. Python's os.stat() function provides access to permissions, ownership (UID/GID on Unix systems), and last-modified timestamps. Permission changes, particularly the addition of world-writable permissions to configuration files or the addition of setuid/setgid bits to executables, are a common indicator of compromise and should be treated as integrity violations of equivalent severity to content modifications.
11.4 Parallelisation and Scalability
For very large directories containing tens of thousands of files, the current sequential scanning approach becomes a performance bottleneck. Python's concurrent.futures.ThreadPoolExecutor would allow multiple files to be hashed simultaneously on multi-core systems. An incremental scanning mode that only re-hashes files whose modification timestamp (mtime) has changed since the last scan would further improve performance for large, slowly-changing file systems, since the majority of files in a stable system will not have changed between scan intervals.
11.5 Remote Baseline Storage and HMAC Protection
The default behaviour of storing the baseline on the same file system being monitored means that a sufficiently privileged attacker who can modify monitored files could also modify the baseline, defeating the integrity assurance. Remote baseline storage  -  for example, using AWS S3 with server-side encryption, or a dedicated secure server  -  would eliminate this attack vector. As an intermediate measure, adding HMAC-SHA256 signing of the baseline file using a user-supplied secret key would detect baseline tampering even if the attacker cannot access the key, providing meaningful protection for deployments where fully remote storage is impractical.
